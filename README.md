# FUEL.GAUGE — AI Handoff Document

This README is written for AI assistants continuing development on this app. Read it fully before touching code.

---

## What This App Is

FUEL.GAUGE is a **personal health OS** — a mobile-first PWA (Progressive Web App) for tracking daily nutrition, exercise, sleep, and macros. The aesthetic is instrument-cluster / telemetry: monospace fonts, dark theme, ring gauges, glowing accents. The app is **single-user per JSONBin account**, login-gated, and syncs to the cloud on every save.

**No build step. No framework. No npm. No bundler.** Pure vanilla JS, HTML, CSS. Open `index.html` in a browser and it runs.

## Architecture

### State

All app data lives in a single `state` object:

```js
{
  profile: {
    name, gender, age, weightKg, heightCm, activityLevel,
    units,           // 'metric' | 'imperial'
    sleepGoalHours   // default 8
  },
  goals: { cal, protein, carbs, fat },
  days: {
    "2026-04-29": {
      food:     [{ id, name, brand?, qty, cal, protein, carbs, fat, ts }],
      exercise: [{ id, name, cal, duration?, ts }],
      sleep:    { bedtimeMin, wakeMin, durationMin }  // optional
    }
  }
}
```

- Date keys are **UTC ISO strings** (`new Date().toISOString().slice(0, 10)`)
- `id` fields are generated by `uid()` — `Math.random().toString(36).slice(2,10) + Date.now().toString(36)`
- `profile: null` means first-time setup not done yet
- `days[key].sleep` is optional — omitted if not logged

### Storage — OPFS + localStorage fallback

`Store` module (`app.js` top):
- Tries **OPFS** (`navigator.storage.getDirectory`) first — faster, no size limit
- Falls back to `localStorage` key `fuelgauge.v1`
- `Store.read()` → parsed state object or `null`
- `Store.write(data)` → persists to whichever backend is available

### Sync — JSONBin.io

`Sync` module (`app.js` top):
- PUT/GET to `https://api.jsonbin.io/v3/b/{bin}` with `X-Master-Key` header
- Credentials stored in `localStorage` as `fuelgauge.syncKey` and `fuelgauge.syncBin`
- `Sync.configure(key, bin)`, `Sync.clear()`, `Sync.isConfigured()`, `Sync.pull()`, `Sync.push(data)`
- **Debounced push**: every `saveState()` call schedules a 1.5s debounced push via `scheduleSyncPush()` — only the latest state fires
- **Merge on startup**: remote wins for `profile`/`goals`; for each day, prefer the source with more food/exercise entries; local sleep is preserved if remote doesn't have it (push may be in-flight)

```js
const saveState = async () => {
  await Store.write(state);
  scheduleSyncPush();
};
```

### Auth

- `users.json` is fetched at login with `{ cache: 'no-store' }` — plain-text password comparison
- On success, session stored in `localStorage` as `fuelgauge.session`: `{ id, jsonbin_key, jsonbin_bin, groq_key }`
- `getSession()` reads it; `applySession()` wires up Sync and stores groq_key
- `doLogout()` clears session + groq_key + Sync credentials, resets state, saves empty state locally, shows login screen
- Login screen: `aria-hidden="false"` shows it, `"true"` hides it (CSS `[aria-hidden="true"] { display:none }`)

### DOM Helpers

```js
const $  = (sel, root = document) => root.querySelector(sel);
const $$ = (sel, root = document) => [...root.querySelectorAll(sel)];
```

Used everywhere. Always prefer these over `document.querySelector` directly.

## Key Patterns

### Toast Notifications

```js
toast(msg, type = 'info')
// type: 'info' (amber), 'success' (green), 'error' (coral), 'loading' (gray, pulsing, no auto-dismiss)
clearToast()  // manually dismiss a loading toast
```

Loading toasts never auto-dismiss — always follow with `toast('...', 'success')` or `clearToast()`.

### Sheets (Bottom Modals)

```js
openSheet('foodModal')
closeSheet('foodModal')
```

HTML: `<div class="sheet sheet--full" id="foodModal" aria-hidden="true">`. CSS `.sheet.open` handles visibility. Sheets support swipe-down-to-close and Escape key. The `[data-close]` attribute on any element inside a sheet closes it.

### Confirm Sheet

Reusable for any destructive action. Uses `pendingDelete` variable:

```js
pendingDelete = async () => { /* do the thing */ };
$('#confirmTitle').textContent = 'Title?';
$('#confirmDesc').textContent  = 'Description.';
$('#confirmYes').textContent   = 'ACTION LABEL';
openSheet('confirmSheet');
```

**Double-confirm pattern** (used for "clear all days"): `pendingDelete` in the first stage sets a new `pendingDelete` and reopens the sheet. Works because `confirmYes` handler captures the fn before nulling: `const fn = pendingDelete; pendingDelete = null; closeSheet(); fn()`.

### Settings Navigation

Stack-based push navigation. `settingsViewStack` tracks history. `navTo(viewId)` slides in a new view, `navBack()` pops. Views are `.settings-view` divs with `active` / `exit-left` CSS classes.

Populate functions are called on navigate: `populateInfoView()`, `populateGoalsView()`, `populateHealthView()`, `populateAppView()`.

### Segmented Controls

All `.seg-btn` clicks are handled by a single delegated listener. Register a callback:

```js
registerSeg('containerId', value => { /* handle */ });
```

### Day Locking

```js
const yesterdayKey = () => { const d = new Date(); d.setUTCDate(d.getUTCDate() - 1); return d.toISOString().slice(0, 10); };
const isPastDay = () => currentDate < yesterdayKey();
```

Today and yesterday are editable. Anything 2+ days ago is locked (view-only). `applyDayLock()` is called from `renderAll()` — it disables the FAB (shows lock icon), disables sleep edit button, shows/hides the cyan ribbon. Delete buttons in food/exercise lists are rendered conditionally based on `isPastDay()`.

**Guard pattern** used on all write actions:
```js
if (isPastDay()) return;
```

### Tab Switching

```js
switchTab('home' | 'exercises' | 'coach')
```

Hides/shows `#homeView`, `#exercisesView`, `#coachView`. Also hides date nav and FAB when not on home. Always calls `closeSettings()` first. Coach tab calls `window.GeminiCoach.onTabOpen()`.

### renderAll

Called after every state change. Renders everything for `currentDate`: hero, rings, sleep, food list, exercise list, week chart, macro chart, streak, date label, day lock state. Never partial-render — always call `renderAll()` after mutations.

## CSS Architecture

### Custom Properties (defined on `:root` per theme)

Key vars: `--bg`, `--bg-2`, `--surface`, `--surface-2`, `--surface-3`, `--ink`, `--ink-soft`, `--ink-mute`, `--ink-faint`, `--line`, `--line-2`, `--amber`, `--green`, `--coral`, `--cyan`, `--sleep`, `--f-display` (Anton), `--f-mono` (JetBrains Mono), `--f-body` (Manrope), `--bar-h` (topbar height).

### Themes

6 themes: `dark`, `light`, `forest`, `ocean`, `ember`, `rose`. Applied via `data-theme` attribute on `<html>`. Saved to `localStorage` as `fuelgauge.theme`.

### Critical CSS Rules

- `.day-lock-ribbon { display: none; }` — **must be `display: none` not `hidden` attribute** — author stylesheet `display: flex` would override the UA `[hidden]` rule. Use `style.display = 'flex'/'none'` from JS.
- `#coachView { top: calc(var(--bar-h) + env(safe-area-inset-top)); }` — desktop media query adds 24px extra for body padding
- `.fab-main:disabled` — lock state styling for past days
- Toast variants: `.toast--success`, `.toast--error`, `.toast--loading`

### Hidden/Shown Pattern

Do **not** use the `hidden` attribute for elements that have `display: flex` or similar set in the stylesheet — the author stylesheet wins. Use `style.display` from JS or `classList.add/remove('hidden')` where `.hidden { display: none !important; }` is defined.

## Coach Module (`coach.js`)

Exposed as `window.GeminiCoach = { onTabOpen }`.

- Uses **Groq API** (OpenAI-compatible): model `llama-3.3-70b-versatile`, streaming SSE
- API key stored in `localStorage` as `fuelgauge.groq_key` (set from session at login)
- `onTabOpen()` checks for key → shows chat or no-key prompt → sends auto brief on first open
- `briefSent` is module-level — resets only on page reload, not re-login within the same session
- `history` array maintains conversation context across messages in the session
- System prompt is rebuilt on every message from `window.FuelGauge.getState()` bridge
- After sending, input is blurred (hides mobile keyboard) — no auto-focus after response

### FuelGauge Bridge (app.js → coach.js)

```js
window.FuelGauge = {
  getState(),
  getCurrentDate(),
  getTotals(date),
  minToTimeStr(min),
  askConfirm(title, desc, action, btnLabel)
}
```

## Calculations

**BMR (Mifflin-St Jeor):**
```
male:   10*weightKg + 6.25*heightCm - 5*age + 5
female: 10*weightKg + 6.25*heightCm - 5*age - 161
```

**TDEE:** `BMR × activityLevel` (1.2 / 1.375 / 1.55 / 1.725 / 1.9)

**Sleep duration:** handles midnight wrap — `wakeMin - bedMin`, add 1440 if ≤ 0.

**Streak:** counts consecutive days with at least one food or exercise entry, working backwards from today. If today has no entries yet, starts from yesterday (today is "in progress").

## Things to Know Before Editing

1. **Always `renderAll()` after mutating state** — there is no fine-grained reactivity
2. **Always `saveState()` after mutating state** — it writes local + schedules cloud push
3. **Date keys are UTC** — use `todayKey()` and `yesterdayKey()` helpers, never `new Date().toLocaleDateString()`
4. **`escapeHtml(str)`** — always escape user-provided strings before inserting into innerHTML
5. **`isPastDay()`** — check this before any write operation when it could be triggered on a past day
6. **Settings views are populated lazily** on navigate — `populateInfoView()` etc. read from `state` each time
7. **The confirm sheet is shared** — always set title/desc/button text before opening it
8. **Loading toasts don't auto-dismiss** — always follow with success/error or `clearToast()`
9. **Service worker caches aggressively** — bump `CACHE_VER` in `sw.js` any time you deploy changes
10. **`coach.js` is loaded after `app.js`** — it relies on `window.FuelGauge` being set, which app.js sets at the bottom of its IIFE
